# AGENTS.md

## このドキュメントの位置づけ

- `AGENTS.md` は、このリポジトリの現状と運用方針を表す北極星ドキュメントとして扱う。
- 仕様変更、運用方針変更、実装判断の前提が変わった場合は、必要に応じてこのファイルを更新する。
- このファイルの記述は、実際のリポジトリ構成と矛盾しないように保つ。

## 現在のリポジトリ構成

- ルートには既存の静的サイト (`index.html`、`contact.php`、`privacy.html`、`thanks.html`) と共有画像素材 (`photo/`) がある。
- ルートの `contact.php` は、LP問い合わせフォームの送信処理として `mb_send_mail()` を使用する。送信元とEnvelope senderは `postmaster@kuroyanagi-clinic.jp`、送信先は `web@kuroyanagi-clinic.jp`、返信先はフォーム入力メールとして扱う。
- `clinic_old/` は作りかけの Next.js 版クロヤナギ医院サイトとして残す。
- `clinic/` は、既存サイトを壊さずゼロベースで作り直す新しいクロヤナギ医院サイトの作業領域として扱う。
- `clinic/` のFVは、PC幅では左上の白い横長ロゴカード、上部の白いナビバー、現行ヒーロースライダー画像を使った大きな写真ステージ、中央寄りの縦書きコピー、左側のSCROLL導線、右側の電話番号ボタン（電話アイコン付き、053-525-1113表示）、および右端の縦型メニューボタンで構成する。角のイラスト装飾は置かない。
- `clinic/` のモバイルFVは、背景写真をヘッダーまで広げ、左右全幅で表示する。左上に白背景のロゴカード、右上にダーク背景のメニューボタンを配置し、キャッチコピーを画面中央付近へ配置する。画面下部には、診療時間・アクセスの固定アクションバーを置く。
- `clinic/` のFVキャッチコピーは、PC/モバイルともに白帯つきの縦書き表現を基準にする。
- `clinic/` のFEATUREセクションは、PC幅では左側の縦書きコピーとタイムラインをsticky表示し、右側では「当院の特徴」をカード化せず導入リードとして常時sticky表示したうえで、4枚の特徴カードだけをスクロールに応じて導入リード下の同じ位置で完全に重ねる。特徴カードは見本画像に寄せた横長カード内に写真・縦ライン装飾・大きな薄い番号・番号つきカテゴリラベル・見出し・本文・説明つきの2つのポイントカードを収める。カードには淡いドット背景、薄い内枠、控えめな区切り線を使い、カードシャドウ、重なりパネル、斜めカット装飾、写真背面の円形アクセントは使わない。最終カードは他カード同様に一定時間stickyで停止させ、その停止位置のまま左側stickyが解除されるタイミングでFEATUREセクション全体が一緒にスクロールする。1180px未満ではsticky演出を解除し、通常の縦積み表示にする。
- `clinic/` のFEATUREセクション左サイドバーは、`FEATURE` ラベル、白帯つき縦書きコピー、タイムラインを一体の章扉風ユニットとして扱う。エディトリアルな読後感を高めるため、ラベルとコピーを繋ぐ物理的な装飾線は廃止し、コンテンツ全体をより高い位置へ配置することで視認性を向上させている。背面には薄い縦長の `FEATURE` タイポを置き、重い影は使わない。
- `clinic/` のFEATUREセクション左側リードコピーは、PC/モバイルとも白帯つきで表示し、文字色は通常のテキストカラーを使う。破線装飾は使わない。
- `clinic/` のNEWSセクションは、PC/モバイルともに「一覧」ボタンを廃止し、左寄せのマガジンスタイルで構成する。
- `clinic/` のFEATUREとSERVICEの間には、三ヶ日の風景写真 `clinic/public/photo/Parallax.jpg` を使ったパララックス区切りセクションを置く。PC幅では固定背景、1024px未満では通常背景として扱う。
- `clinic/` のFACILITYセクション直下には、併設施設「みっかび東」をさらっと紹介し、`https://kuroyanagi-clinic.jp/` へ遷移する関連施設導線を置く。FACILITYスライダーの補足導線として見えるよう、淡いブルーグレー背景の上に中央寄せのコンパクトな濃いブルーの横長カードを置き、右側の角丸写真はカード内に収める。
- Cloudflare Workers へのデプロイは、ルートの `wrangler.jsonc` で `clinic/out` を静的アセットとして指定する。
- ルートの `requirements_definition.md`、`requirements.md`、`design.md`、`tasklist.md` は、現在の `clinic/` 新規構築方針を管理する。

## ステアリングファイル運用ルール

このプロジェクトでは、要件・設計・実装タスクを以下のファイルに分割して管理する。

### 1. `requirements_definition.md`

- プロダクト要件定義書として扱う。
- 記載内容:
  - プロダクトビジョンと目的
  - ターゲットユーザーと課題・ニーズ
  - 主要な機能一覧
  - 成功の定義
  - ユーザーストーリー
  - 受け入れ条件
  - 機能要件
  - 非機能要件

### 2. `requirements.md`

- 今回の作業単位での要求内容を整理する。
- 記載内容:
  - 変更・追加する機能の説明
  - ユーザーストーリー
  - 受け入れ条件
  - 制約事項

### 3. `design.md`

- 今回の変更内容の設計を整理する。
- 記載内容:
  - 実装アプローチ
  - 変更するコンポーネント
  - データ構造の変更
  - 影響範囲の分析

### 4. `tasklist.md`

- 具体的な実装タスクと進捗管理に使う。
- 記載内容:
  - 実装タスク
  - タスクの進捗状況
  - 完了条件

## 運用ポリシー

- `AGENTS.md` は、プロジェクトの現状を表す北極星ドキュメントとして扱う。
- 機能追加、仕様変更、運用方針変更が発生した場合、Codex は必要に応じて `AGENTS.md` を更新し、記述を最新状態に保つ。
- 要件整理、仕様確認、実装判断が必要な作業では、まず `requirements_definition.md` を参照する。
- `requirements_definition.md` が存在しない場合は、必要に応じて新規作成する。
- `AGENTS.md` と `requirements_definition.md` に差分が生じた場合は、両方を同時に更新して整合を保つ。
- 作業単位で詳細な変更要件が必要な場合は `requirements.md` を作成または更新する。
- 実装前に設計の明文化が必要な場合は `design.md` を作成または更新する。
- 実装を分解して進捗を管理する場合は `tasklist.md` を作成または更新する。
- まだ存在しないファイルについては、運用が始まるまでは「今後この構成で運用する」前提として扱い、すでに運用中であるとは記述しない。

## 作業時の判断基準

- リポジトリの構造や仕様の前提を変える変更では、コード修正だけでなく `AGENTS.md` 更新要否も確認する。
- 大きな機能追加や画面改修では、必要に応じて `requirements.md`、`design.md`、`tasklist.md` を追加する。
- 実装内容とドキュメントの整合を崩さないことを優先する。
- `clinic/` の作業では、既存ルートサイトと `clinic_old/` を参照対象として扱い、完成前に置き換えたり削除したりしない。
